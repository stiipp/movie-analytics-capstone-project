## 3.1 Architecture Overview

This project uses a local, Docker-based modern data stack to turn raw movie files into a reporting-ready warehouse for Power BI.

The pipeline is built around a medallion architecture:

- **Bronze:** raw landing and enrichment outputs
- **Silver Base:** cleaned canonical tables
- **Gold Prep:** exploded mapping tables for many-to-many relationships
- **Gold:** dimensional and analytical marts

This separation keeps ingestion, cleaning, dimensional modeling, and reporting logic distinct.

## 3.2 End-to-End Flow

The pipeline flow is:

`Raw Files -> Bronze -> Silver Base -> Gold Prep / Silver -> Gold -> Power BI`

Execution order in Airflow:

1. `extract_raw`
2. `ingest_raw`
3. `spark_enrich`
4. `clean_pandas`
5. `spark_transform`
6. `dbt_build`

## 3.3 Source Layer

Primary local inputs:

- `data/raw/project_data.zip`
- `data/raw/TMDB_movie_dataset_v11.csv`

Processed local outputs:

- `data/processed/movies_main.csv`
- `data/processed/movie_extended.csv`
- `data/processed/ratings.json`

This project is intentionally local-only. Optional API-based ingestion was removed so the workflow stays simpler and easier to reproduce.

## 3.4 Warehouse Layers

### Bronze Layer

Purpose:
- land raw source data with minimal transformation
- preserve source state before business logic is applied

Main tables:
- `bronze.movies_main`
- `bronze.movie_extended`
- `bronze.ratings`
- `bronze.ext_tmdb_raw`
- `bronze.movies_enriched`
- `bronze.bronze_enrichment_audit`

### Silver Base Layer

Purpose:
- standardize the raw data into reliable movie-grain and attribute-grain tables

Main tables:
- `silver_base.movies`
- `silver_base.ratings`
- `silver_base.movie_extended`

Key responsibilities:
- deduplicate movie IDs
- parse dates
- clean numeric fields
- derive financial-quality flags
- flatten ratings into relational form

### Gold Prep Layer

Purpose:
- create bridge-ready exploded mappings from cleaned extended attributes

Main tables:
- `gold_prep.movie_genres`
- `gold_prep.movie_companies`
- `gold_prep.movie_countries`

These tables are generated by `transform_spark.py` from `silver_base.movie_extended`.

### Silver Layer

Purpose:
- expose stable dbt staging views over `silver_base` and `gold_prep`

Main models:
- `silver.stg_movies`
- `silver.stg_movies_roi_eligible`
- `silver.stg_ratings`
- `silver.stg_movie_extended`
- `silver.stg_movie_genres`
- `silver.stg_movie_companies`
- `silver.stg_movie_countries`

### Gold Layer

Purpose:
- provide business-facing dimensions, bridges, and analytical marts for downstream reporting

Core dimensional models:
- `gold.dim_movie`
- `gold.dim_date`
- `gold.dim_budget_tier`
- `gold.fact_movie_performance`

Reusable supporting dimensions:
- `gold.dim_genre`
- `gold.dim_company`
- `gold.dim_country`

Bridge tables:
- `gold.bridge_movie_genre`
- `gold.bridge_movie_company`
- `gold.bridge_movie_country`

Analytical marts:
- `gold.fact_movie_roi_analysis`
- `gold.fact_movie_market_coverage`
- `gold.fact_genre_budget_roi`
- `gold.fact_genre_budget_roi_recommendable`
- `gold.fact_genre_market_momentum`
- `gold.fact_genre_market_coverage_trends`

## 3.5 Core Architectural Pattern

The reporting layer is intentionally centered on a movie-grain fact table:

- `fact_movie_performance`

That core fact is supported by:

- `dim_movie`
- `dim_date`
- `dim_budget_tier`

This is the main star schema used by the dashboard. Genre, company, and country are modeled separately through bridge tables so the warehouse remains reusable if the analytical focus shifts later.

## 3.6 Business Logic by Layer

- **Bronze:** preserve raw data and enrichment inputs
- **Silver Base:** clean and standardize the source data
- **Gold Prep:** explode multivalue attributes into row-level mappings
- **Silver:** expose reusable staging views
- **Gold:** apply business semantics for ROI analysis, market coverage, trend analysis, and reporting

## 3.7 Why Medallion Was Used

The medallion structure makes this project easier to reason about and defend:

- raw data stays separate from corrected business-ready data
- cleaning logic is isolated from reporting logic
- dbt models can rely on cleaner upstream inputs
- downstream tools like Power BI consume stable, curated tables instead of raw files

For a capstone project, this also makes the architecture easier to explain during presentation because each layer has a clear purpose.

## 3.8 Reporting Architecture

Power BI is meant to consume the `gold` layer rather than recreate major business rules in DAX. The main reporting inputs are:

- `gold.fact_movie_performance`
- `gold.fact_movie_roi_analysis`
- `gold.fact_movie_market_coverage`
- `gold.fact_genre_budget_roi`
- `gold.fact_genre_budget_roi_recommendable`
- `gold.fact_genre_market_momentum`
- `gold.fact_genre_market_coverage_trends`
- supporting dimensions and bridges

This keeps the semantic model clearer and pushes the heavier logic into the warehouse, where it is easier to document and test.
